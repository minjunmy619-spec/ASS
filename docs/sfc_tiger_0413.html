    本质上，这不是“再改模型”，而是把内存分层做清楚：

    最理想的做法是这三层分开：

    1. DSP SRAM
    - 例如 kv_cache、conv cache、hidden state

    2. NPU 权重区 / 设备 DDR / Flash 映射区
    - 放 band_bias、decode_basis、routing_bias、expansion_basis 这类常量
    - 这些在设备启动时加载一次，后面每帧复用

    3. 每帧输入输出
    - 只传 x 和 prev_state
    - 输出 y 和 next_state

    所以“改成设备侧非-DSP常驻资源方案”，通常有三种实现路径。

    **方案 A：最好**
    把这些常量当成模型权重/initializer，直接编进 NPU 编译产物。
    这样运行时它们是“模型静态参数”，不算 DSP 的 per-frame state 预算。

    适用条件：
    - 你的 NPU 编译器能接受这些 initializer

    **方案 B：次优**
    如果编译器对 ONNX 内嵌常量敏感，就把它们导出成“外部静态输入”，但在设备端只加载一次、长期绑定。
    也就是：
    - 推理时：每帧不重新生成，只复用同一块 buffer handle

    这时虽然它们在图接口上是 input，但语义上是“常驻静态资源”，不是“每帧动态输入”。

    **方案 C：最差**
    如果目标 runtime 会把所有 graph input 每次都拷进 DSP SRAM，那外置成 input 也没意义。
    这种情况下必须退回方案 A，或者进一步缩小这些常量。

    对我们当前 repo 来说，已经做了第一步准备：
    - --externalize-band-constants 已经能把这些常量从 initializer 挪成显式输入
    - 这适合做“设备端静态绑定”
    - 但它还不等于“自动不占 DSP”，最终取决于你的 NPU runtime 怎么放置 input buffer

      per_frame():
        prepare x
        prepare prev_state
        run model(x, prev_state, const_handles...)
        get y, next_state

    - state 是“每帧传递”

    - 模型从输入开始就只认 512

    - 模型本身更小
    - ONNX 更干净
    - streaming state 确实会变小
    - 因为模型内部的第一层 causal cache 已经只按 512 走，不再按 1025
    - 更利于 NPU 编译
    - 设备侧更灵活
    - 你可以先固定前端，后面换不同 online family 共用同一个 512-bin frontend

    **代价**
    - 训练和部署必须严格一致
    - 训练时也必须喂这个 preprocessing 后的频谱
    - 如果输出还是 mask 到原始 1025-bin 频谱，就需要一个对应的 postprocessing
    - 512 -> 1025 的展开
    - 这样一来，模型学到的是“压缩频谱空间上的分离”，不是原始 STFT 空间上的直接分离

    当这件事是：
    - 固定的
    - 无状态的
    - 不需要学习的
    - 想被多个模型共享的

    要看你设备链路里，preprocessing 是放在哪。

    如果 preprocessing 在 **NPU 外面**，例如 CPU / DSP 前处理：
    - 这是非常合理的方案
    - 甚至通常比塞进模型里更好

    如果 preprocessing 也必须塞进 ONNX/NPU 图里：
    - 那它本质上还是 model 的一部分，只是语义上叫 preprocessing
    - 这时收益主要是结构清晰，不是部署边界变化

    **结论**
    - **可以做成 NPU 友好的 cross attention**
    - 但前提是它是 **按帧做的频率维 cross attention**，不是时间维 self-attention
    - 1x1 Conv2d
    - Softmax
    - 基础 tensor op

    **为什么“有可能”**
    你 repo 里的论文版 CrossAttnEncoder / CrossAttnDecoder，本质是：

    - 每一帧把 F 个频率 token 当作 KV
    - 做一次 QK^T -> softmax -> AV

    这件事本身不依赖未来时间帧，所以对 online 来说不是问题。
    它的问题主要是**实现形式**，不是数学形式。

    当前实现里不友好的点在：
    - scaled_dot_product_attention
    - Linear
    - 比较重的 position bias 处理

    这些都不适合你的 NPU 约束。

    **怎么改成 NPU 友好**
    可以改写成下面这种等价形式：

    1. Q/K/V 投影改成 1x1 Conv2d
    - 原来 Linear(d, d) 的地方，改成对 (B, C, T, F) 的 Conv2d(kernel=1)
    - 这和线性层是等价的，只是张量布局不同

    2. attention logits 用 bmm
    - 对每个 frame 做：
    - Q: (B*T, K, D)
    - K: (B*T, F, D)
    - A = bmm(Q, K^T) 得到 (B*T, K, F)
    - decoder 则是 (B*T, F, K)

    3. 加静态 band mask / position bias
    - 如果要保论文味道，可以加 band-local bias
    - 最稳的是把它做成：
    - 或部署时作为 persistent constant input
    - 不建议运行时动态生成复杂 mask

    4. softmax
    - 在最后一个维度上做即可
    - 这个在你们现在 ONNX allowlist 里已经是允许的

    5. attention 输出再用 bmm
    - Y = bmm(A, V)
    - encoder 得 (B*T, K, D)
    - decoder 得 (B*T, F, D)

    6. 输出投影仍用 1x1 Conv2d

    **和你现在约束的关系**
    这版是能贴住你约束的：

    - tensor 维度可以控制在 4 维以内
    - 所以它不会增加 streaming state 预算
    - online 的 cache 仍主要来自 causal conv separator，而不是 attention 本身

    **真正的代价**
    代价主要不是 state，而是**算力和中间激活**：

    - encoder attention 复杂度大约是 O(T * K * F * D)
    - decoder attention 复杂度大约是 O(T * F * K * D)
    - 当 F=1025 时会偏重
    - 如果先用了你前面加的 frequency preprocessing，把 1025 -> 512，就会好很多

    所以现实建议是：
    - **如果要做 NPU-friendly cross attention，最好基于 F=512 或更低的前处理版本做**
    - 否则算力和带宽压力会比较大
