    1. 保留顶层接口形态
    输入继续用现在的 subband_spec_RIs，STFT/ISTFT 仍放在模型外部，这一点是对的。

    - subband feature extraction

    当前 BNBlock 用的是 LayerNorm + Conv2d，建议改成更贴近官方的：

    - PReLU
    - Conv2d(feature_dim, bw * 4 * num_sources, kernel_size=1, groups=num_sources)

    3. Recurrent 主体保持原始迭代结构
    保留原始这条链：
    - x
    - freq_path
    - residual add
    - frame_path
    - residual add
    - 多次 iteration，中间经过 concat_block(mixture + x)

    不要改成另一种 block family。当前 [StatefulTimeUConvBlock](/home/cmj/works/spectral-feature-compression/TIGER/tiger_online.py:2056) 已经不是原始 UConvBlock 的在线化了，更像替代结构。

    4. FreqUConvBlock 可以保留 2D 化，但 TimeUConvBlock 要改成“原始 UConvBlock 的 causal chunk 版”
    建议新做一个 CausalTimeUConvBlock2D，它本质上还是原始 UConvBlock 的 2D 对应物：

    - proj_1x1
    - spp_dw[0]: Conv2d(k=(1,5), stride=(1,1))
    - spp_dw[1:]: Conv2d(k=(1,5), stride=(1,2))
    - globalatt: (1,5) depthwise conv
    - loc_glo_fus
    - last_layer: 保持 kernel 5，不要改成 1

    在线化方式不要再保存 3 组内部卷积 state，而是改成“块输入上下文”模式：

    - 每个 iteration 只缓存该 block 的输入上下文 ctx_i
    - 当前 chunk 输入前拼接 ctx_i
    - 整块跑完后，只截取最后 chunk_T 的有效输出
    - next_ctx_i = 当前 block 输入的最后 R 帧

    这比现在 state_0/state_1/state_2 更 faithful，也更容易和离线数学对齐。

    5. Attention 保留 TIGER 的 asym 设计，但做成静态维度版本

    - nband 和 chunk_T 必须在 __init__ 里显式传入
    - qk_dim = hid_channels * nband
    - 训练和部署都使用同一个 sliding-window 定义
    - streaming 分支必须有 valid-length mask，避免冷启动零缓存被参与 softmax

    如果要尽量保留 TIGER 原始 stride=2 的 U-shape，多半就不该做 T=1 的单帧推理。更合理的是固定小 chunk，比如：

    - chunk_T = 4 或 8
    - frame attention window = W
    - 每次 ONNX 输入一个 chunk
    - 输出同长度 chunk
    - 外部 runtime 维护 context 和 KV

    **状态设计**

    1. time_ctx

    - [B, num_blocks * out_channels, nband, ctx_len]

    2. frame_kv
    每个 iteration 一个 attention cache
    形状统一成：
    - [B, n_heads, kv_window, num_blocks * (qk_dim + v_dim)]

    这样导出时仍只有固定几个 tensor 输入输出，但语义比现在清晰很多。

    **训练方式**

    最稳的是训练时就按 online chunk 仿真：

    - 随机切 chunk 序列
    - 前向时显式传递 time_ctx 和 frame_kv
    - loss 只算每个 chunk 的有效输出
    - 可用离线官方 TIGER 做 teacher，做 warm start 或蒸馏

                  self.prev_states_2 = new_s2

                  y_block = frame_time[:, : self.hop]                 # [num_sources, 512]

    - 如果时间卷积是 causal 的
    - padding 只在左边补
    - 输出时刻 t' 只由输入中不晚于该时刻的样本决定

    那么 stride=2 仍然是 causal 的。

    比如时间维上做：

    第 n 个输出只会看输入区间里截止到当前的历史，不会看未来。
    所以从“因果性”本身看，stride>1 是允许的。

    但在 streaming 部署里，问题不只是 causal，还包括 **在线调度和状态管理**。
    stride>1 会带来这几个额外麻烦：

    1. 输出不是每帧都产生
    比如 stride=2，上层每来 2 个时间步才更新 1 次。
    这意味着多尺度路径会变成不同“时钟”。

    2. 状态更难维护
    你不仅要存左上下文，还要处理：
    - 哪一层这一步该不该更新
    - 没更新时上一次低速层特征怎么复用
    - 上采样路径怎么和当前高分辨率步对齐

    3. 训练和部署更容易不一致
    离线时 stride=2 很自然；
    但在线单帧 T=1 时，如果不专门设计 multirate state machine，很容易和离线结果不一致。

    所以结论是：

    - **从数学上说：可以，仍然能 causal**
    - **从工程上说：可以，但 streaming 实现复杂很多**

    在你的场景里，这就变成一个取舍：

    **方案 A：保留 stride>1**
    - 更接近原始 TIGER 的 UConv 设计
    - 多尺度感受野更像原模型

    **方案 B：去掉时间路径里的 stride>1**
    - 用 causal dilated conv 或 stride=1 的多层局部块替代
    - 更容易做 1-frame online
    - 更容易导出到 NPU
    - 但会偏离原始 TIGER 一些

    结论先说：

    - **是的，这种“block 输入上下文 ctx_i”方案可以明显优化 state 设计**
    - 它的最大优点不是一定绝对最小，而是：
    - state 语义更干净
    - 更接近离线 TIGER 的整块卷积

    但也要分清一个点：

    - 如果 chunk_T = 1，ctx_i 方案不一定比当前 3 组 state 小很多
    - 如果 chunk_T > 1，ctx_i 方案通常更有优势，也更自然

    **为什么它可能更省**

    你现在这版 time path 是 3 层 causal dilated conv：

    - 第 1 层需要历史宽度 R0
    - 第 2 层需要历史宽度 R1
    - 第 3 层需要历史宽度 R2

    所以当前每个 iteration 要存：

    - state_0: [B, C, nband, R0]
    - state_1: [B, C, nband, R1]
    - state_2: [B, C, nband, R2]

    这本质上是在缓存 **每个内部子层的输入历史**。

    而你说的 ctx_i 方案是：

    - 对每个 iteration，只缓存“该整个 time block 的输入上下文”
    - 下次推理时，把 ctx_i 和当前 chunk 在 block 输入端拼起来
    - 整个 block 一次算完
    - 最后只取末尾 chunk_T 的有效输出

    这样 state 会变成：

    - ctx_i: [B, C_block_in, nband, R_block]

    如果 R_block 不比 R0 + R1 + R2 更大太多，那就有机会比三组 state 更省，而且接口会更干净。
